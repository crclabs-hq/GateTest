# Performance Benchmark — 2026-05-15

Real numbers, run locally against the GateTest repository itself (102 modules total available).

## Setup

- **Target:** GateTest repo (`/home/user/GateTest`), ~146 files, ~1.5 MB of source
- **Host:** Sandbox dev environment, Node 22.22.2
- **Command:** `time node bin/gatetest.js --suite <suite>`
- **No caching, no parallelization config changes**

## Results

| Suite | Modules run | Wall time | Issues found | Verdict for customer-facing use |
|---|---|---|---|---|
| **quick** | 40 | **22.4 seconds** | 1 syntax error, 2 lint errors, 2 error-swallows, 14 env-var-missing, ~600 warnings | ✅ Fast enough for HN "try it now" buyer-confidence |
| **full** | 66 | **7m 37s** | 27 SQL-injection findings, 2 soft-fail-gate CI issues, 2 error-swallows, 14 env-vars, 1 mutation baseline, ~1100 warnings | ❌ TOO SLOW for serverless. Vercel function cap is 300s. |

## Honest interpretation

### Quick scan = launch-ready

22 seconds against our own repo is a great number. A customer landing on `gatetest.ai`, pasting their repo URL, and watching a progress bar fill for ~30 seconds is a fine HN-grade experience. They'll wait.

This validates the Free Preview + $29 Starter / $49 Health Check tier UX. **No action needed.**

### Full scan = needs work before going live as a paid tier

7 minutes locally exceeds Vercel's 300-second function cap. Three reasons it was slow on this run:

1. **dataIntegrity module** found 27 SQL-injection candidates — each requires AST traversal across `src/`, `lib/`, and `website/`. Cumulative cost is significant on a 146-file repo.
2. **Universal-language-checker modules** (python, go, rust, java, ruby, php, csharp, kotlin, swift) all loop through every source file looking for language-specific patterns. With 9 of them, that's 9 passes of file iteration per scan.
3. **`mutation`** module is computationally heavy by design — it forks the code, applies operators, would normally invoke a test runner if available.

### What it WON'T be on production Vercel

Production scans through `scan/run/route.ts` already cap `MAX_FILES_TO_READ = 50` (we cap the in-memory file map). So a real customer scan touches at most 50 files, not 146. That alone cuts the per-scan time roughly in half.

Plus production skips:
- `e2e`, `visual`, `chaos` (need Playwright + a deployed URL)
- `aiReview` (only runs at $399 Nuclear)
- `mutation` (only runs locally via CLI, not the website route)

**Realistic production estimate for a typical customer repo (~50 files):**
- Quick: 8-15 seconds
- Full (without e2e/visual/chaos/mutation/aiReview): 60-120 seconds
- Nuclear (adds aiReview + correlation + diagnoser): 120-240 seconds with Claude API time dominating

All inside the 300s Vercel cap.

## Things found about our own repo that we should fix before HN

The full-scan output flagged real issues in our own code that an HN reader running our CLI would see:

1. **27 SQL-injection candidates** in our own modules — these are largely false positives because they're scanner module CODE that DETECTS SQL injection (the patterns include the literal string `SELECT * FROM ${var}` which is what we look FOR in customer code). dataIntegrity module needs a self-exemption pattern. **Self-fix needed before launch** so customers running `gatetest --suite full ./` on us don't see us flagged for SQL-injection.

2. **2 soft-fail-gate violations** in `.github/workflows/ci.yml:59` and `:97` — these are `continue-on-error: true` on the gate step itself. **Bible Forbidden #24 violation.** Already flagged previously; should be cleared.

3. **14 env-var-missing entries in `.env.example`** — all the new env vars we added today (GATETEST_INPUT_USD_PER_MTOK, etc.). Easy fix: update `.env.example` with the new variable names.

4. **2 error-swallows** — `AIBuilderHandoff.tsx:168` and `cli-engine-runner.js:265`. Both legitimate (genuine fallback paths) but flagged. Should add `// error-ok` markers.

5. **1 markdown lint error** in `docs/ops/hn-launch-pitch-draft.md` — already on the FixList.
